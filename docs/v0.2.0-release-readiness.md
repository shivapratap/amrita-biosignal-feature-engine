# ABFE v0.2.0 integration and release-readiness design

This document defines the controlled path from the audited
`codex/complexity-design` branch to the ABFE v0.2.0 GitHub Release. Passing a
gate permits work on the next gate; it does not authorize merging, pushing,
tagging, or publishing. Those external actions require separate explicit
approval.

## Candidate scope

Version 0.2.0 adds the approved fractal/complexity feature family:

1. Lempel-Ziv complexity;
2. Hjorth mobility;
3. Hjorth complexity;
4. Fisher information;
5. Petrosian fractal dimension;
6. Katz fractal dimension;
7. Higuchi fractal dimension;
8. detrended fluctuation analysis;
9. Rosenstein largest Lyapunov exponent.

The first eight scalar features participate in default extraction. Largest
Lyapunov exponent remains request-only because scientifically meaningful
embedding, delay, temporal-separation, fit-interval, and sampling-rate
parameters cannot be inferred safely.

The scientific implementation phase is closed at remediation commit
`e02f88e84959f016702c2f210110af64478a2d8c`. Its focused independent re-audit
returned `PASS WITH OBSERVATIONS`: both findings were resolved, 131,068 binary
sequences produced no LZ76 mismatch, and the corrected duplicate-embedding
Lyapunov result agreed with an independent oracle within approximately
`1.1e-16`.

## Release principles

- Freeze the audited formulas, defaults, units, degeneracy policies, registry
  policy, and public result contracts unless a demonstrated defect is found.
- Give every numerical correction before/after evidence and a focused
  regression test under `CONTRIBUTING.md`.
- Keep NumPy and SciPy as the only runtime dependencies.
- Keep AntroPy, DIHC, nolds, MNE, and sampen-profile pinned validation
  authorities rather than runtime dependencies.
- Build and publish only from the exact immutable commit tagged `v0.2.0`.
- Never move or replace an existing tag.
- Keep independent audit reports and imported auditor instructions out of the
  public repository and release artifacts.
- Preserve GitHub Releases as the only approved distribution channel for
  v0.2.0 unless the user separately approves PyPI.

## Batch A: integration inventory and repository hygiene

1. Compare the complete feature branch against the published v0.1.0 commit and
   classify every changed file as implementation, test, validation authority,
   documentation, benchmark, CI, or release metadata.
2. Confirm the intended nine-feature scope and reject unrelated source changes.
3. Verify that caches, virtual environments, build products, local audit
   reports, imported auditor instructions, and credentials are neither tracked
   nor included by the artifact builder.
4. Confirm all reference dependencies are exactly pinned in CI.
5. Record the focused complexity re-audit disposition without committing the
   external audit report.
6. Resolve the pre-existing untracked `AGENTS.md` separately; do not silently
   delete or publish it.

Exit evidence: reviewed branch diff, clean tracked-file inventory, pinned
authority inventory, and explicit disposition of every untracked file.

## Batch B: user documentation and public API synchronization

1. Update the README feature catalogue, counts, examples, validation summary,
   performance description, and links for all nine complexity features.
2. Update `docs/api-reference.md` with signatures, parameters, return units,
   exceptions, `NaN` conditions, top-level exports, default membership, and
   request-only Lyapunov extraction.
3. Ensure `docs/complexity-validation.md` remains the normative numerical
   specification and agrees with production.
4. Extend `examples/api_smoke_test.py` to demonstrate representative direct
   complexity calls and `LargestLyapunovRequest` without presenting a positive
   exponent as proof of deterministic chaos.
5. Run all documented examples against the development source and later
   against an installed artifact.
6. Update citation and release notes only when the final release date and
   version are frozen.

Exit evidence: complete API catalogue, executable examples, documentation link
check, and no stale v0.1.0-only capability or feature-count claims.

## Batch C: evidence and performance closure

1. Retain the independently reproduced LZ76 numerical equivalence and scaling
   evidence.
2. Strengthen `benchmarks/results/lz76-remediation.json` with:
   - parent and remediation commit hashes;
   - deterministic binary-input digests;
   - explicit clock, warm-up, repeat, and aggregation metadata;
   - peak-memory observations or a documented auxiliary-space analysis.
3. Add a direct Lyapunov sampling-frequency scaling regression test so the
   inverse-seconds unit contract is explicit.
4. Run the full deterministic benchmark harness from an installed development
   artifact and record the environment.
5. Treat timings as observational evidence, not machine-specific release
   thresholds.
6. Do not begin unrelated entropy or PSD optimization during release
   stabilization.

Exit evidence: provenance-complete remediation benchmark, explicit unit-scaling
test, installed-artifact benchmark run, and unchanged numerical compatibility.

## Batch D: CI, artifacts, and release-workflow generalization

1. Run the supported Python and dependency-floor CI matrix, Ruff, strict mypy,
   cache rejection, and the complete offline suite.
2. Run the pinned external-reference job in its clean Python 3.12 CI
   environment, including all complexity and pre-existing reference modules.
3. Build wheel and source distribution from a clean checkout.
4. Validate metadata and contents with `scripts/check_artifacts.py`.
5. Install the wheel and source distribution into separate fresh environments
   outside the checkout and run `scripts/smoke_installed.py`.
6. Verify the complexity module, typed marker, license, documentation, and all
   required package sources are present while audit material and caches are
   absent.
7. Generalize `.github/workflows/release.yml` so its non-publishing manual
   dry-run validates the current candidate rather than requiring v0.1.0.
8. Preserve the safety boundary: manual dispatch has read-only permissions and
   cannot publish; only a pushed version-matching `v*` tag can enter the
   publishing job.
9. Run the generalized dry-run remotely and verify that the publishing job is
   skipped.

Exit evidence: green CI, pinned reference results, validated artifacts, two
fresh-install smoke results, and a successful non-publishing workflow dry-run.

## Batch E: final independent release-candidate audit

Provide the exact still-development-version candidate to a new independent
read-only auditor. Require review of:

- all nine formulas and declared invariances;
- minimum lengths, parameter validation, and `NaN` degeneracy behavior;
- LZ76 exhaustive-history equivalence and scaling;
- Lyapunov duplicate handling, units, and request-only orchestration;
- registry order, default membership, extractor dispatch, diagnostics, and
  immutable provenance;
- public exports, documentation examples, and feature counts;
- supported Python/dependency floors and pinned references;
- wheel/sdist contents and clean installations;
- benchmark evidence provenance;
- CI and tag-gated release-workflow safety;
- absence of audit files, caches, secrets, and unrelated changes.

Every Blocker, High, or release-relevant Medium finding must be resolved and
rechecked. Lower-severity findings require a written disposition.

Exit evidence: independent `PASS` or `PASS WITH OBSERVATIONS` with no unresolved
release-blocking findings.

## Batch F: version freeze and integration

This batch requires explicit approval.

1. Change `0.2.0.dev0` to `0.2.0`.
2. Convert the unreleased changelog into a dated `0.2.0` release entry and
   finalize `CITATION.cff`.
3. Rerun documentation, quality, full reference, benchmark, artifact, and
   clean-install gates on the exact frozen commit.
4. Review the feature-branch history and integrate it into `main` using the
   explicitly approved merge strategy.
5. Push the reviewed `main` commit and require all remote CI jobs to pass.
6. Do not create the tag during this batch.

Exit evidence: exact approved release commit on public `main`, frozen version
metadata, green remote CI, and recorded commit hash.

## Batch G: publication and post-publication verification

This batch requires a separate explicit approval after Batch F passes.

1. Create annotated tag `v0.2.0` on the exact approved `main` commit.
2. Verify the tag target locally before pushing only that tag.
3. Monitor the tag-triggered GitHub Release workflow to completion.
4. Verify the public release URL, artifact filenames, and `SHA256SUMS`.
5. Download the published wheel into a fresh environment, install it, and run
   the installed-package smoke test from outside a checkout.
6. Confirm `pip install .` still works from a clean clone.
7. If any publication step fails, do not move or replace the tag; stop and
   report the failure.

Exit evidence: immutable public v0.2.0 GitHub Release, verified hashes, and
successful installation of the downloaded artifact.

## Independent-audit observations and disposition

The focused remediation re-audit left three non-blocking observations:

- benchmark commit and input hashes: accepted into Batch C;
- explicit Lyapunov sampling-frequency regression: accepted into Batch C;
- LZ-specific peak-memory evidence: accepted into Batch C.

The auditor could not build artifacts because its sandbox lacked network
access. ABFE's primary development run independently built, inspected, and
fresh-installed both artifacts from commit `e02f88e`; Batch D will repeat these
checks in clean CI and from the final candidate.

## Approval boundaries

Approval of this design authorizes documentation and local validation work
within the named batches. It does not authorize:

- deletion of untracked external files;
- changing the frozen numerical contract;
- adding runtime dependencies;
- merging into `main`;
- pushing commits;
- creating or pushing a tag;
- publishing a GitHub Release;
- publishing to PyPI.

Each external or irreversible transition remains separately approved.
